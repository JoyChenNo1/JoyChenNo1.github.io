Go 并发编程篇（五） 条件变量 Condition

发表于2023-12-16|更新于2025-03-18|golang

前言

鸟窝的《深入理解 Go 并发编程》读书笔记以及其他go并发编程知识学习的汇总，持续完善中。


Cond

用于等待某个条件的一组 goroutine， 当条件变为 true 时，其中一个或者一组 goroutine 被唤醒。使用场景较少，使用 chan 可以代替 Cond 的功能，更为地道。

使用方法

package main

import (
    "fmt"
    "sync"
)

// Task 定义任务类型
type Task func()

// ThreadPool 线程池结构体
type ThreadPool struct {
    tasks    chan Task
    wg       sync.WaitGroup
    cond     *sync.Cond
    mutex    sync.Mutex
    stopped  bool
}

// NewThreadPool 创建一个新的线程池
func NewThreadPool(size int, capacity int) *ThreadPool {
    pool := &ThreadPool{
       tasks: make(chan Task, capacity),
    }
    pool.cond = sync.NewCond(&pool.mutex)
    for i := 0; i < size; i++ {
       pool.wg.Add(1)
       go pool.worker()
    }
    return pool
}

// worker 工作 goroutine
func (p *ThreadPool) worker() {
    defer p.wg.Done()
    for {
       p.mutex.Lock()
       // 当任务队列为空且线程池未停止时，等待任务
       for len(p.tasks) == 0 && !p.stopped {
          p.cond.Wait()
       }
       // 如果线程池已停止且任务队列为空，退出循环
       if p.stopped && len(p.tasks) == 0 {
          p.mutex.Unlock()
          break
       }
       // 从任务队列中取出一个任务
       task := <-p.tasks
       p.mutex.Unlock()
       // 执行任务
       task()
    }
}

// Submit 提交任务到线程池
func (p *ThreadPool) Submit(task Task) {
    p.mutex.Lock()
    // 如果线程池已停止，不接受新任务
    if p.stopped {
       p.mutex.Unlock()
       return
    }
    p.tasks <- task
    // 通知一个等待的工作 goroutine 有新任务
    p.cond.Signal()
    p.mutex.Unlock()
}

// Shutdown 关闭线程池
func (p *ThreadPool) Shutdown() {
    p.mutex.Lock()
    p.stopped = true
    // 通知所有等待的工作 goroutine 线程池已停止
    p.cond.Broadcast()
    p.mutex.Unlock()
    // 等待所有工作 goroutine 完成任务
    p.wg.Wait()
    close(p.tasks)
}

func main() {
    // 创建一个包含 3 个工作 goroutine，任务队列容量为 5 的线程池
    pool := NewThreadPool(3, 5)
    // 提交一些任务
    for i := 0; i < 5; i++ {
       index := i
       pool.Submit(func() {
          fmt.Printf("Task %d is running\n", index)
       })
    }
    // 关闭线程池
    pool.Shutdown()
    fmt.Println("ThreadPool is shut down")
}

实现

// Cond implements a condition variable, a rendezvous point
// for goroutines waiting for or announcing the occurrence
// of an event.
//
// Each Cond has an associated Locker L (often a [*Mutex] or [*RWMutex]),
// which must be held when changing the condition and
// when calling the [Cond.Wait] method.
//
// A Cond must not be copied after first use.
//
// In the terminology of [the Go memory model], Cond arranges that
// a call to [Cond.Broadcast] or [Cond.Signal] “synchronizes before” any Wait call
// that it unblocks.
//
// For many simple use cases, users will be better off using channels than a
// Cond (Broadcast corresponds to closing a channel, and Signal corresponds to
// sending on a channel).
//
// For more on replacements for [sync.Cond], see [Roberto Clapis's series on
// advanced concurrency patterns], as well as [Bryan Mills's talk on concurrency
// patterns].
//
// [the Go memory model]: https://go.dev/ref/mem
// [Roberto Clapis's series on advanced concurrency patterns]: https://blogtitle.github.io/categories/concurrency/
// [Bryan Mills's talk on concurrency patterns]: https://drive.google.com/file/d/1nPdvhB0PutEJzdCq5ms6UI58dp50fcAN/view
type Cond struct {
  noCopy noCopy

  // L is held while observing or changing the condition
  L Locker

  notify  notifyList
  checker copyChecker
}

Broacast 允许调用者唤醒所有等待此 Cond 的 goroutine。调用后会清空等待队列，把等待的 goroutine 全部唤醒，对应 C++ 的 condition_variable.notify_all().

func (c *Cond) Broadcast() {
  c.checker.check()
  runtime_notifyListNotifyAll(&c.notify)
}

Signal 允许调用者唤醒一个等待此 Cond 的 goroutine。对应 C++ 的 condition_variable.notify_one().

// Signal() does not affect goroutine scheduling priority; if other goroutines
// are attempting to lock c.L, they may be awoken before a "waiting" goroutine.
func (c *Cond) Signal() {
  c.checker.check()
  runtime_notifyListNotifyOne(&c.notify)
}

// Wait atomically unlocks c.L and suspends execution
// of the calling goroutine. After later resuming execution,
// Wait locks c.L before returning. Unlike in other systems,
// Wait cannot return unless awoken by [Cond.Broadcast] or [Cond.Signal].
//
// Because c.L is not locked while Wait is waiting, the caller
// typically cannot assume that the condition is true when
// Wait returns. Instead, the caller should Wait in a loop:
//
//	c.L.Lock()
//	for !condition() {
//	    c.Wait()
//	}
//	... make use of condition ...
//	c.L.Unlock()
func (c *Cond) Wait() {
  c.checker.check()
  t := runtime_notifyListAdd(&c.notify)
  c.L.Unlock()
  runtime_notifyListWait(&c.notify, t)
  c.L.Lock()
}

使用规则

1、调用 Cond.Wait() 时，一定要加锁：

c.L.Lock()
c.Wait()
c.L.Unlock()

2、唤醒之后必须要检查判断条件，条件是否满足需要调用 Wait 的 goroutine 负责。
