Go 并发编程篇（1） Go 并发编程和调度器

发表于2023-12-15|更新于2025-03-19|golang

前言

鸟窝的《深入理解 Go 并发编程》读书笔记以及其他 go 并发编程知识学习的汇总，持续完善中。


并发 vs 并行

并发的本质是要设计一个结构，可以使程序的不同计算模块并发地执行。这些模块的执行，对于单核 CPU，并发地执行（不能并行地执行），在一定的时间段内，他们都可以占用 CPU 时间片，每个模块都可以被执行；对于多核 CPU，这些模块可能真的是并行的。总之，并发的目标之一就是能利用并行（多核）的能力，但是并行的目标并不是并发。

我们在设计并发程序的时候，经常要进行并发单元的设计，并且要进行并发模块之间的数据同步和消息传递，甚至要编排任务，让它们按照固定的流程执行。


并发编程的优化是有上限的

提升系统的一部分性能对整个系统的性能有多大影响？根据阿姆达尔定律:

S = \frac{1}{ (1-α) + α/k}

假设原来系统执行一个程序需要的时间为 T, 其中某一部分占用的时间百分比为 α, 如果这一部分的性能提升 k 倍，则整个系统执行此程序需要的时间会变为：

T' = \text{不能并发执行的部分耗时} + \text{并发提升后所需耗时}

加速比（系统性能提升的倍数 T/T’）：

S = \frac{1}{ (1-α) + α/k}

在设计并发程序的时候，尽量让可并发的部分在整个系统中占较大（α ↑），得到更大加速比。尽量减少串行部分，k ↑。


Go 并发不一定最快

使用一个快排程序进行对比，并行不一定快过串行的情况：

package quicksort

func quick_sort(a []int, start, end int) {
  if start > end {
    return
  }

  p := partition(a, start, end)
  quick_sort(a, start, p-1)
  quick_sort(a, p+1, end)
}

func partition(a []int, start, end int) int {
  // 将最后一个值作为分界值
  pivot := a[end]
  i := start - 1
  for j := start; j < end; j++ {
    // 如果小于分界值，挪到左边
    if a[j] < pivot {
      i++
      a[j], a[i] = a[i], a[j]
    }
  }
  a[i+1], a[end] = a[end], a[i+1]
  return i + 1
}
package quicksort

func ConcurrencyQuickSort(a []int, start, end int, done chan<- struct{}) {
  if start >= end {
    done <- struct{}{}
    return
  }

  p := partition(a, start, end)
  childDone := make(chan struct{}, 2)
  // 启动一个 goroutine，快排左边
  go ConcurrencyQuickSort(a, start, p-1, childDone)
  // 启动一个 goroutine，快排右边
  go ConcurrencyQuickSort(a, p+1, end, childDone)

  <-childDone
  <-childDone
  done <- struct{}{}
}
package quicksort

func ImprovedConcurrencyQuickSort(a []int, start, end int, 
                                  done chan struct{}, depth int) {
  if start >= end {
    done <- struct{}{}
    return
  }

  depth--
  p := partition(a, start, end)
  if depth > 0 {
    childDone := make(chan struct{}, 2)
    // 启动一个 goroutine，快排左边
    go ImprovedConcurrencyQuickSort(a, start, p-1, childDone, depth)
    // 启动一个 goroutine，快排右边
    go ImprovedConcurrencyQuickSort(a, p+1, end, childDone, depth)

    <-childDone
    <-childDone
  } else {
    QuickSort(a, start, p-1)
    QuickSort(a, p+1, end)
  }

  done <- struct{}{}
}

func QuickSort(a []int, start, end int) {
  if start > end {
    return
  }

  p := partition(a, start, end)
  QuickSort(a, start, p-1)
  QuickSort(a, p+1, end)
}
package main

import (
  "fmt"
  "main/quick_sort"
  "math/rand"
  "time"
)

func main() {
  benchmark()
}

func benchmark() {
  rand.Seed(time.Now().UnixMicro())
  var (
    n         = 10000000
    testData1 = make([]int, 0, n)
    testData2 = make([]int, 0, n)
    testData3 = make([]int, 0, n)
  )

  for i := 0; i < n; i++ {
    val := rand.Intn(n * 100)
    testData1 = append(testData1, val)
    testData2 = append(testData2, val)
    testData3 = append(testData3, val)
  }

  func() {
    timeNow := time.Now()
    defer func() {
      fmt.Println(`串行快排耗时:`, time.Since(timeNow))
    }()
    quick_sort.QuickSort(testData1, 0, len(testData1)-1)
  }()

  func() {
    timeNow := time.Now()
    defer func() {
      fmt.Println(`并行快排耗时:`, time.Since(timeNow))
    }()

    done := make(chan struct{})
    go quick_sort.ConcurrencyQuickSort(testData2, 0, len(testData2)-1, done)
    <-done
  }()

  func() {
    timeNow := time.Now()
    defer func() {
      fmt.Println(`优化过的并行快排耗时:`, time.Since(timeNow))
    }()

    done := make(chan struct{})
    go quick_sort.ImprovedConcurrencyQuickSort(testData3, 0, len(testData3)-1, done, 3)
    <-done
  }()
}
./benchmark          
串行快排耗时: 826.007458ms
并行快排耗时: 2.669086625s
优化过的并行快排耗时: 480.616209ms

结论： Go1.19 后会根据 goroutine 栈的使用率来初始化新的 goroutine 栈的大小，goroutine 的栈大小不再是固定的 2KB。因为创建新的 goroutine 会伴随着栈的分配，损耗性能；同时，大量 goroutine 在调度和垃圾回收检查时会占用一定时间。所以并发设计上，需要控制 goroutine 的数量，充分利用并发优势的同时又可以减少管理太多 goroutine 带来的损耗。


Go 运行时调度器（G-M-P）

上下文切换（context switch）

系统会在中断、系统调用时执行线程上下问切换，线程上下文切换是一种昂贵的操作，操作系统需要将用户态->内核态，保存要切换线程的执行状态，将一些重要寄存器的值和进程状态保存在线程控制块数据结构中。当恢复线程运行时，需要将这些状态加载到寄存器中，从内核态->用户态。

goroutine 调度是由 Go 运行时控制的，每个编译的 Go 程序都会附加一个很小的 Go 运行时，负责内存分配、goroutine 调度和卡机回收。goroutine 会和某个线程绑定，它是用户态的，并且初始的栈比较小。所以它的上下文切换比较小，大致上，可以认为 goroutine 的上下文切换开销是线程上下文切换开销的十分之一。

G-M-P 模型

G M P
表示 goroutine，存储与 goroutine 相关的信息，如栈、状态、要执行的函数等。 表示执行计算资源单元，Go 会把它和操作系统的线程对应，只有 P 和 M 绑定之后，才能让 P 的本地队列中的 goroutine 运行起来。 表示逻辑 processor，非指 CPU 的处理器！！！ P 负责把 M 和 G 捏合起来，让一系列的 goroutine 在某个 M 上顺序执行。默认情况下，NumberOf(P) = CPU 逻辑数。I/O敏感场景下，可以使用 runtime.GOMAXPROCS 改变它。

G-M-P 模型示意图


运行 Go 程序时：

1、会调用 Go 运行时的一个函数完成运行时的初始化工作（调度器初始化、垃圾回收等）。最开始会创建 m0 和 g0，完成调度器初始化的工作，为 main.main 生成一个 goroutine，并且被 m0 执行。

2、Go 运行时会启动 n 个 P 和 M，并且把 P 和 M 捏合在一起，除非有阻塞的 I/O 导致 P 和 M 解绑， P 再找到新的 M。

对于每个 P：

（1）都有自己的本地队列，他会顺序执行它本地的 goroutine 队列；

（2）每 61 次或者本地队列没有可执行的 goroutine 时，会从全局队列找到一个 goroutine 来执行，避免全局队列 goroutine 没有机会执行；

（3）它也会从其他的 goroutine 队列窃取一半的 goroutine 过来，让工作均衡。

3、对于 timer 和 netpoll：

timer 经过几次演化，它的四叉堆依附在 P 上， P 在调度的时候，会检查这个四叉堆上面是否有 timer 需要触发，同时也会窃取 timer。

调度器也会检查 netpoll，对于那些网络 I/O 已经就绪的 goroutine，也会有机会执行 read/write 操作。

Symmon 是一个独立的 goroutine，不依附在任何 P 上，而是运行在一个独立的 M 中，定时运行一次，检查与网络 I/O 相关的 goroutine和那些长时间运行（>10 ms）的 goroutine，避免某个 goroutine 长时间占用计算资源。

Go 1.14 之后，实现了基于信号的抢占式调度，向正在运行的 goroutine 所绑定的 M 发出 SIGURG 信号，信号处理函数会进行调度，让其他 goroutine 有机会执行。

m0 和 g0 是两个特殊的对象：

m0 g0
m0 是 Go 程序启动时的第一个线程，也是主线程。
m0 在全局变量中，其他一般的 M 都是运行时的局部变量。		 g0 是调度用的 goroutine， 每个 M 都只有一个 g0。
g0 的栈是系统分配的，在 Linux 上栈大小默认固定为 8MB，不能扩展和缩小。
而通常的 goroutine 栈的大小默认是 2KB（Go 1.19 改成了根据历史使用自动调整的方式），它们的栈大小可以扩展。
一个 goroutine 运行完成后，后者新建一个 goroutine 时，调度器会先被切换到 g0 上，让 g0 负责调度。

对于 m0 绑定的那个 g0，也是放在全局变量中的。

GMP 模型中协程的调度和分配本质就是一种percpu 模式，均匀且充分的进行分配，达到高效的并发执行。
